# Signed URL Management Specialist

## When to Use
- Implementing file upload flows that use presigned PUT URLs (direct-to-storage uploads)
- Implementing file download flows that use presigned GET URLs (time-limited access)
- Reviewing existing signed URL generation for expiry, scope, or leakage issues
- Designing a signed URL rotation or revocation strategy
- Troubleshooting signed URL authentication errors (403, InvalidSignature)

---

## Step 1 — Classify the Use Case

| Use Case | URL Type | Method | Notes |
|---|---|---|---|
| Browser uploads file directly to storage | Presigned PUT | `PUT` | Never route bytes through API server |
| Server uploads file | Server-side SDK call | — | No presigned URL needed |
| User downloads private file | Presigned GET | `GET` | Short TTL, generated on demand |
| CDN serves file publicly | Public bucket URL | `GET` | Only for truly public assets |
| Resumable large upload | Multipart presigned URLs | `PUT` per part | SDK-managed (Transfer Manager) |

## Step 2 — Presigned PUT (Upload URLs)

### Generation
```go
// Go — aws-sdk-go-v2
presigner := s3.NewPresignClient(s3Client)
req, err := presigner.PresignPutObject(ctx, &s3.PutObjectInput{
    Bucket:      aws.String(bucket),
    Key:         aws.String(objectKey),   // server-generated UUID key
    ContentType: aws.String(contentType), // enforce type at signing time
}, s3.WithPresignExpires(15*time.Minute)) // short TTL for upload
```

```python
# Python — boto3
url = s3_client.generate_presigned_url(
    "put_object",
    Params={"Bucket": bucket, "Key": object_key, "ContentType": content_type},
    ExpiresIn=900,   # 15 minutes
)
```

```typescript
// Node.js — @aws-sdk/s3-request-presigner
import { getSignedUrl } from "@aws-sdk/s3-request-presigner";
const url = await getSignedUrl(s3Client, new PutObjectCommand({
  Bucket: bucket,
  Key: objectKey,
  ContentType: contentType,
}), { expiresIn: 900 });  // 15 minutes
```

**Upload URL Rules:**
- [ ] TTL: **10–15 minutes** — long enough for the client to upload, short enough to limit replay
- [ ] `ContentType` locked at signing time — prevents type-switching on upload
- [ ] `ContentLengthRange` set via conditions where supported (limits max upload size)
- [ ] Object key is a new UUID generated by the server per request — never reuse
- [ ] Persist `status = 'pending'` in DB before returning the URL to the client

## Step 3 — Upload Confirmation Flow

The signed URL completes the upload but your application must verify it:

```
1. Client: POST /api/upload-url            → Server returns { upload_url, object_key }
2. Client: PUT <upload_url> (bytes)        → Storage (no server involvement)
3. Client: POST /api/uploads/confirm       → { object_key }
4. Server: HeadObject(object_key)          → verifies object exists and gets size/ETag
5. Server: UPDATE storage_objects SET status='ready' WHERE object_key=...
6. Server: Publish event → downstream processing queue
```

- [ ] Always `HeadObject` to confirm upload before marking as `ready`
- [ ] Reject confirmation requests for `object_key`s not in `status = 'pending'` for the authenticated tenant
- [ ] Set a TTL job to purge `status = 'pending'` objects older than 1 hour (abandoned uploads)

## Step 4 — Presigned GET (Download URLs)

```go
req, err := presigner.PresignGetObject(ctx, &s3.GetObjectInput{
    Bucket:                     aws.String(bucket),
    Key:                        aws.String(objectKey),
    ResponseContentDisposition: aws.String(`attachment; filename="report.pdf"`),
}, s3.WithPresignExpires(5*time.Minute))
```

**Download URL Rules:**

| Scenario | Recommended TTL |
|---|---|
| User viewing a document inline | 5–10 minutes |
| User downloading a file | 5–15 minutes |
| Shared link (user shares with colleague) | See Step 5 |
| API response embedding the URL | 1–2 minutes (URL for immediate use) |
| CDN behind presigned URL | 1–24 hours (CDN caches, revalidates) |

- [ ] Never store presigned GET URLs in the database — generate on demand per request
- [ ] Generate per-user, per-request to prevent link sharing leaking access
- [ ] Set `ResponseContentDisposition` to control inline vs. download behavior
- [ ] Set `ResponseContentType` to prevent MIME sniffing attacks

## Step 5 — Shareable Links (Long-lived Access)

When users need to share a link with colleagues **outside the normal session flow**:

**Do NOT** generate a presigned URL with a 7-day TTL. Instead:

```
1. Create a `share_tokens` record in the DB:
   - token: random 32-byte hex (crypto-random, not UUID)
   - object_id: FK to storage_objects
   - created_by: user_id
   - expires_at: now() + 7 days (configurable)
   - revoked_at: nullable

2. Serve via your API:
   GET /share/{token}
   → Validate token (not expired, not revoked)
   → Generate fresh presigned GET URL (5 min TTL)
   → Redirect 302 to presigned URL
```

**Why:** This pattern allows revocation, audit logging, and rate limiting — impossible with raw presigned URLs.

- [ ] Share tokens are cryptographically random, not UUIDs (no sequential enumeration)
- [ ] Revocation supported (`revoked_at`)
- [ ] Expiry enforced in DB before generating the presigned redirect
- [ ] Download events logged to audit table

## Step 6 — Security Checklist

- [ ] Presigned URLs generated only for objects belonging to the authenticated tenant
- [ ] Object key validated against `storage_objects` table before signing (no key guessing)
- [ ] Upload URLs: `ContentType` bound at signing time
- [ ] Download URLs: generated per-request, never cached server-side or stored in DB
- [ ] No presigned URL with TTL > 24 hours
- [ ] `STORAGE_SECRET_ACCESS_KEY` never logged, never returned in API responses
- [ ] HTTPS enforced — presigned URLs must not be served over HTTP
- [ ] For shareable links: token-based indirection with server-side revocation support

---

## Quality Checks

- [ ] Upload flow: client POSTs to API for URL, uploads directly to storage, confirms to API
- [ ] `HeadObject` confirmation runs before marking upload `ready`
- [ ] Presigned GET URLs have TTL ≤ 15 min for user-facing downloads
- [ ] Shareable links use server-side token indirection, not long-TTL presigned URLs
- [ ] No presigned URL derived from user-supplied key — always server-controlled object key
- [ ] CORS configured to accept `PUT` from allowed origins for upload URLs
- [ ] Audit trail for share link creation, access, and revocation
