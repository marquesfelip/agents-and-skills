# SBOM Generation Specialist

## When to Use

Invoke this skill when you need to:
- Generate a Software Bill of Materials (SBOM) for an application or container image
- Choose between CycloneDX and SPDX formats
- Integrate SBOM generation into a CI/CD pipeline
- Store, sign, and distribute SBOMs alongside release artifacts
- Use SBOMs for vulnerability management and license compliance tracking
- Meet regulatory or contractual SBOM requirements (EO 14028, NTIA, SOC 2)

---

## Step 1 — Understand SBOM Fundamentals

An SBOM is a machine-readable inventory of all components in a software artifact:

**NTIA Minimum Elements (required for compliance):**
| Element | Description |
|---|---|
| Supplier name | Who created the component |
| Component name | Name of the dependency |
| Version | Exact version string |
| Other unique identifiers | Package URL (purl), CPE |
| Dependency relationship | Which component depends on which |
| Author of SBOM data | Who generated this SBOM |
| Timestamp | When the SBOM was generated |

**SBOM Formats:**
| Format | Standard Body | Output | Best For |
|---|---|---|---|
| **CycloneDX** | OWASP | JSON, XML | Security tooling, VEX, attestation |
| **SPDX** | Linux Foundation | JSON, YAML, RDF, tag-value | Legal/license compliance, open source |

Checklist:
- [ ] SBOM format chosen based on consumer requirements (security tooling → CycloneDX; legal → SPDX)
- [ ] NTIA minimum elements confirmed to be present in generated SBOM
- [ ] SBOM scope defined: source only, compiled artifact, or container image?

## Step 2 — Choose SBOM Generation Tool

| Tool | Ecosystems | Output Formats | Notes |
|---|---|---|---|
| **Syft** (Anchore) | Multi-ecosystem, container images, filesystems | CycloneDX, SPDX, Syft JSON | OSS, fast, CI-friendly |
| **Trivy** | Same as Syft + IaC | CycloneDX, SPDX | Already used for vuln scanning; one tool |
| **cdxgen** | Multi-ecosystem (deep Go, Java, Node support) | CycloneDX | Best CycloneDX fidelity |
| **CycloneDX tools** | Language-specific plugins | CycloneDX | Ecosystem-native accuracy |
| **SPDX tools** | SDK-based generation | SPDX | For SPDX-specific requirements |

**Recommendation:**
- Use **Syft** or **Trivy** for container image SBOMs (simplest CI integration)
- Use **cdxgen** for source-level SBOMs with full dependency graph fidelity

Checklist:
- [ ] Tool chosen generates SBOMs at the correct scope (source vs image)
- [ ] Tool supports the required output format (CycloneDX or SPDX)
- [ ] Tool produces `purl` (Package URL) identifiers for each component — enables CVE correlation

## Step 3 — Generate SBOM in CI

**Syft — filesystem SBOM:**
```bash
# Install Syft
curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh -s -- -b /usr/local/bin

# Generate CycloneDX SBOM from source
syft dir:. -o cyclonedx-json=sbom.cdx.json

# Generate SBOM from a Docker image
syft myapp:${{ github.sha }} -o cyclonedx-json=sbom-image.cdx.json
```

**Trivy — SBOM from image:**
```bash
trivy image --format cyclonedx --output sbom.cdx.json myapp:${{ github.sha }}
```

**GitHub Actions integration:**
```yaml
- name: Generate SBOM (source)
  uses: anchore/sbom-action@v0
  with:
    format: cyclonedx-json
    output-file: sbom.cdx.json

- name: Generate SBOM (container image)
  uses: anchore/sbom-action@v0
  with:
    image: myapp:${{ github.sha }}
    format: spdx-json
    output-file: sbom-image.spdx.json

- name: Upload SBOM as release asset
  uses: actions/upload-artifact@v4
  with:
    name: sbom
    path: "*.json"
```

Checklist:
- [ ] SBOM generated from the **final built artifact** (image or binary) — not just source files
- [ ] Both source SBOM and image SBOM generated (different component sets)
- [ ] SBOM stored as a CI artifact tied to the commit SHA
- [ ] SBOM attached to GitHub Release assets for each release tag

## Step 4 — Sign and Attest the SBOM

Signing proves the SBOM was generated by a trusted pipeline and has not been tampered with:

**Cosign attestation (SLSA provenance + SBOM):**
```bash
# Sign the SBOM and attach it to the image as an attestation
cosign attest \
  --predicate sbom.cdx.json \
  --type cyclonedx \
  myapp:${{ github.sha }}
```

**GitHub Actions with keyless signing (Sigstore):**
```yaml
- name: Sign SBOM with Cosign (keyless)
  uses: sigstore/cosign-installer@v3

- name: Attest SBOM to image
  run: |
    cosign attest \
      --yes \
      --predicate sbom.cdx.json \
      --type cyclonedx \
      ${{ env.IMAGE_URI }}@${{ steps.build.outputs.digest }}
```

**Verify an attestation:**
```bash
cosign verify-attestation \
  --type cyclonedx \
  --certificate-identity-regexp "https://github.com/myorg/myrepo" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
  myapp:latest
```

Checklist:
- [ ] SBOM signed with Cosign using keyless signing (Sigstore Fulcio/Rekor) — no key management needed
- [ ] Attestation attached to the container image in the registry
- [ ] Verification command documented in the runbook and tested
- [ ] Signature verification integrated into deployment pipeline — reject unsigned images

## Step 5 — Store and Distribute SBOMs

**Storage options:**
| Location | Mechanism | Use Case |
|---|---|---|
| OCI Registry (attached to image) | Cosign attestation / OCI referrers | Verified alongside image pull |
| GitHub Release assets | Upload on tag | Human-readable, customer-accessible |
| Artifact store (S3, GCS) | Upload with SHA-based path | Long-term archival |
| Dependency Track | SBOM upload API | Centralized vulnerability and license management |

**Dependency Track integration:**
```bash
# Upload SBOM to Dependency Track for continuous vulnerability tracking
curl -X POST "https://dependencytrack.example.com/api/v1/bom" \
  -H "X-Api-Key: $DT_API_KEY" \
  -F "project=$PROJECT_UUID" \
  -F "bom=@sbom.cdx.json"
```

Checklist:
- [ ] SBOM stored in at least two locations: attached to image (Cosign) + release asset
- [ ] Dependency Track (or equivalent) configured to ingest SBOMs for ongoing vulnerability monitoring
- [ ] Historical SBOMs retained per release for at least 3 years (compliance)
- [ ] SBOM generation timestamp and pipeline run ID embedded in the SBOM metadata

## Step 6 — Use SBOMs for Vulnerability Management

With a machine-readable SBOM, vulnerability correlation becomes automated:

```bash
# Scan an SBOM with Grype (instead of rescanning the artifact)
grype sbom:sbom.cdx.json

# Or with Trivy
trivy sbom sbom.cdx.json
```

**VEX (Vulnerability Exploitability eXchange):**
- A VEX document states whether a CVE in the SBOM actually affects the product
- Use when a library has a CVE but the vulnerable code path is not used
- CycloneDX supports embedded VEX statements

Checklist:
- [ ] SBOM used as input for vulnerability scanning — avoids re-scanning the artifact
- [ ] VEX documents created for CVEs confirmed non-exploitable in context
- [ ] SBOM-based scanning integrated into the weekly security triage process
- [ ] New CVEs published against known components detected via SBOM correlation (Dependency Track alerts)

## Output Report

### Critical
- No SBOM generated — regulatory requirement (EO 14028) potentially unmet
- SBOM not generated from the final artifact — source SBOM misses OS-level and image dependencies
- SBOM not signed — consumers cannot verify provenance or integrity

### High
- SBOM generated but not stored alongside the release — not accessible for audit
- NTIA minimum elements missing (no purl, no dependency relationships) — SBOM is not standards-compliant
- No automated CVE correlation against SBOM — vulnerability window not detected

### Medium
- Only source SBOM generated — image SBOM missing OS packages and runtime dependencies
- SBOM not consumed by Dependency Track or equivalent — generated but not used for ongoing monitoring
- VEX not used — non-exploitable CVEs inflate the finding count and waste triage time

### Low
- SBOM format not aligned to consumer requirements (SPDX delivered to security tooling expecting CycloneDX)
- Historical SBOMs not retained — cannot audit past releases
- SBOM generation adds > 2 min to CI — Syft/Trivy not configured with parallelism or caching

### Passed
- SBOM generated from final built image using Syft or Trivy on every release
- CycloneDX SBOM signed with Cosign and attached to the OCI image as an attestation
- SBOM stored as a release asset and in Dependency Track for ongoing monitoring
- NTIA minimum elements present; purl identifiers included for all components
- VEX documents used to suppress non-exploitable CVEs with documented justification
